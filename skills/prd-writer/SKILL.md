# PRD文档书写助手

## 概述

本Skill帮助用户按照标准化四步流程编写高质量的PRD（产品需求文档）。

## 使用流程

当用户需要创建PRD时，按照以下四个步骤引导完成：

### 第一步：背景、目标、用户画像

引导用户提供以下信息：

1. **项目背景**
   - 为什么要做这个项目？
   - 当前面临的问题或机会是什么？
   - 市场/业务背景是什么？

2. **项目目标**
   - 核心目标（1-3个）
   - 可量化的成功指标（KPI）
   - 预期上线时间

3. **用户画像**
   - 目标用户群体描述
   - 用户痛点
   - 用户需求场景

**输出格式示例：**
```markdown
## 1. 项目背景
[背景描述]

## 2. 项目目标
- 核心目标：[目标描述]
- 成功指标：[可量化指标]
- 预期上线：[时间]

## 3. 用户画像
### 用户群体：[群体名称]
- **特征**：[描述]
- **痛点**：[痛点描述]
- **使用场景**：[场景描述]
```

### 第二步：功能列表

基于第一步的内容，梳理功能需求：

1. **核心功能**（必须实现）
2. **扩展功能**（锦上添花）
3. **未来规划**（后续版本）

每个功能项包含：
- 功能名称
- 功能描述
- 优先级（P0/P1/P2）
- 备注

**输出格式示例：**
```markdown
## 4. 功能列表

### 4.1 核心功能（P0）
| 功能名称 | 功能描述 | 优先级 | 备注 |
|---------|---------|-------|------|
| [功能名] | [描述] | P0 | [备注] |

### 4.2 扩展功能（P1）
[同上格式]

### 4.3 未来规划（P2）
[同上格式]
```

### 第三步：用户故事（Gherkin语言）

使用Gherkin语法编写用户故事，格式如下：

```gherkin
Feature: [功能名称]
  As a [角色]
  I want [需求]
  So that [价值]

  Scenario: [场景名称]
    Given [前置条件]
    When [操作步骤]
    Then [预期结果]
    And [额外结果]
```

**编写要点：**
- 每个核心功能对应一个Feature
- 每个Feature包含多个Scenario覆盖不同场景
- 使用Given-When-Then结构清晰描述
- 包含正常流程和异常流程

**输出格式示例：**
```markdown
## 5. 用户故事

### 5.1 [功能名称]

```gherkin
Feature: [功能名]
  As a [角色]
  I want [需求]
  So that [价值]

  Scenario: [正常场景]
    Given [前置条件]
    When [操作]
    Then [结果]

  Scenario: [异常场景]
    Given [前置条件]
    When [错误操作]
    Then [错误提示]
```
```

### 第四步：验收标准

为每个功能定义清晰的验收标准：

1. **功能验收标准**
   - 功能是否完整实现
   - 边界条件处理
   - 错误处理机制

2. **非功能验收标准**
   - 性能要求
   - 兼容性要求
   - 安全要求

3. **测试验收标准**
   - 单元测试覆盖率
   - 集成测试通过标准
   - 用户验收测试(UAT)标准

**输出格式示例：**
```markdown
## 6. 验收标准

### 6.1 [功能名称]验收标准

#### 功能验收
- [ ] 标准1：[描述]
- [ ] 标准2：[描述]

#### 非功能验收
- [ ] 性能：[要求]
- [ ] 兼容性：[要求]

#### 测试验收
- [ ] 单元测试覆盖率≥[X]%
- [ ] UAT通过率100%
```

## 完整PRD模板

```markdown
# [项目名称] PRD文档

> 版本：[版本号]  
> 日期：[日期]  
> 作者：[作者]

---

## 1. 项目背景
[内容]

## 2. 项目目标
[内容]

## 3. 用户画像
[内容]

## 4. 功能列表
[内容]

## 5. 用户故事
[内容]

## 6. 验收标准
[内容]

## 7. 附录
- 术语表
- 参考文档
- 变更记录
```

## 使用指导

1. **逐步引导**：一次只进行一个步骤，完成后再进入下一步
2. **提问澄清**：每个步骤开始前，先询问用户已有信息，补充缺失内容
3. **即时整理**：每完成一步，立即整理输出该部分内容
4. **最终汇总**：四步完成后，汇总生成完整PRD文档
5. **格式规范**：使用Markdown格式，保持结构清晰

## 注意事项

- 保持语言简洁明了
- 避免技术实现细节，聚焦"做什么"而非"怎么做"
- 用户故事要可测试、可验证
- 验收标准要具体、可量化
- 及时保存进度，避免信息丢失
