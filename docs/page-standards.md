# 页面标准

## 通用页面骨架

后台页面默认由四层组成：

1. `PageHeader`：页面标题、业务说明和右侧主操作。
2. `filter-panel`：查询条件，使用 `Form layout="inline"`。
3. `SectionPanel`：承载表格、详情、配置区块和状态反馈。
4. Drawer 或 Modal：承载详情补充、编辑、审批确认和危险操作。

页面主间距统一为 20px，内容容器圆角不超过 8px。表格页面默认支持横向滚动，避免字段压缩导致信息不可读。

## 列表页标准

- 适用：客户、订单、审批、合同、组织成员等数据管理页面。
- 标准结构：查询区、批量操作区、表格、分页。
- 普通数据列表页不展示页面头、说明文案、指标卡或提醒条。若确有待办提醒，优先沉入状态列、行高亮或批量操作区，避免在查询区上方形成第二套操作入口。
- 查询条件不展示字段名前缀，使用输入框、选择器和日期组件的提示词表达字段含义。
- 查询、重置和页面主操作放在同一行，动作紧跟查询条件之后。
- 查询动作和业务动作必须分组展示。按钮按重要性从左到右排列，高重要性按钮居左，辅助和工具动作靠后。`查询` 是查询区唯一蓝色主按钮，`重置` 使用普通按钮；新建类主业务动作使用成功色浅底描边样式；导入、导出、批量处理等工具按钮使用中性浅底样式，并与查询动作之间保留分隔线或明显间距。按钮颜色必须引用全局色板，不在页面中新增局部色值。
- 表格固定列只用于编号和操作列。
- 金额、数量等数字列右对齐。
- 状态统一用 `Tag`，颜色按业务语义分配。
- 操作列宽度按实际可见动作估算，不预留过宽空白。只有字段很多且横向滚动明显影响操作效率时，才固定右侧操作列。
- 操作列必须使用 `TableActions`，不得在表格行内自行平铺多个按钮。操作项 2 个及以下时直接展示；达到 3 个及以上时只展示前 2 个高频动作，剩余动作统一折叠进右侧 `更多`。业务页面不得自定义折叠数量或自定义按钮绕过该折叠规则。动作顺序按使用频率和业务重要性排列，危险动作如不属于前 2 个高频动作，默认放入 `更多` 并使用危险色。行内普通操作和 `更多` 使用同一全局链接动作色。

## 左树右表页标准

- 适用：部门树、分类树、组织树、区域树等分层对象驱动的数据列表。
- 标准结构：左侧树面板、右侧查询区、右侧表格、分页。
- 左侧树使用通用 `TreeListLayout` 承载，树面板内顶部固定为节点搜索，下面直接展示树组件。
- 树下方不展示“当前节点”“节点数量”等摘要块；当前选中态由树节点高亮表达，节点相关信息放入右侧列表或详情弹窗。
- 根节点可在右侧展示新增图标，用于新增一级或下级节点。
- 普通节点的新增、编辑、删除统一收纳在节点右侧操作入口中，默认 hover 或选中时展示。
- 删除节点必须二次确认；业务页面如存在引用关系，需要在删除前补充引用校验。
- 右侧列表必须继承列表页标准，包含序号列、查询区、标准分页和 `TableActions` 操作列。

## 详情页标准

- 适用：客户详情、合同详情、审批详情、人员详情。
- 标准结构：对象标题、关键动作、指标摘要、基础信息、关联业务、侧栏动态。
- 主区承载核心数据，侧栏只放联系人、动态、风险提醒等辅助信息。
- 关联数据使用 Tabs，不在一个页面中连续堆叠过多表格。

## 表单页标准

- 适用：新建、编辑、提交审批、配置录入。
- 标准结构：标题动作、步骤条、分组表单、附件、提交动作。
- 表单分组使用 `SectionPanel`，每组字段保持两列布局，小屏自动单列。
- 必填校验在字段级声明，提交前不做额外说明页。

## 配置页标准

- 适用：审批规则、权限角色、租户参数、通知策略。
- 如果页面主体是配置对象管理，例如模板、规则、参数集合，默认先按列表页标准构建：查询区、表格、分页。复杂编辑、预览和发布校验进入 Drawer、Modal 或独立详情页。
- 只有当页面目标明确是单个配置对象的编辑时，才使用配置分类 Tabs、左侧基础配置、右侧规则或成员列表。
- 列表页下方不常驻配置工作台。常驻工作台会和列表主任务竞争空间，除非该路由被明确设计为工作台页。
- 开关类配置使用 `Switch`，枚举配置使用 `Select`，规则集合使用 Table。
- 保存和发布动作放在页面头部右侧。

## 审批页标准

- 通用审批查询能力必须包含：流程名称、单据号、发起人、审批摘要、创建时间。
- 已办审批需要额外包含办理类型，枚举为已通过、已驳回、已转办；办理类型表示当前用户对该单据的办理动作，不等同于单据状态。
- 已办审批列表需要展示单据状态和结果，状态表示单据进行中或已结束，结果表示处理中、已通过、已驳回或已撤回。
- 抄送给我不展示办理类型。
- 我发起的审批需要包含：流程名称、单据号、审批摘要、状态、结果、发起人、创建时间。
- 审批详情必须展示审批详情信息和审批链路。
- 审批摘要统一格式为 `{{字段名1}}：{{字段值1}} | {{字段名2}}：{{字段值2}}...`。
- 审批详情不单独展示审批摘要，摘要只在列表中展示。
- 待办审批列表不展示当前节点；已办审批和抄送给我的当前节点展示当前审批人的用户姓名，或者流程结束。
- 待办审批详情允许通过、驳回、转办，操作固定在详情底部，并提供审批意见输入框；驳回时审批意见必填，未填写需要弹窗提示补充。
- 批量通过、批量驳回属于同一类业务动作，按钮形态保持一致；批量通过使用正向语义色，批量驳回使用危险语义色。
- 审批详情右上角需要提供全屏展示操作。
- 已办审批、抄送给我、我发起的详情默认只读。
- 我发起的审批中，状态为进行中时允许撤回；已结束不可撤回。
- 我发起的列表中，进行中单据需要同时展示详情和撤回两个操作。

## 状态页标准

- 空数据：使用 `Empty`，描述具体业务对象。
- 无权限：使用 `Result status="403"`，提供申请权限入口。
- 系统异常：使用 `Result status="500"`，提供重试动作。
- 加载中：局部区域使用 `Spin`，不阻断整个应用框架。
