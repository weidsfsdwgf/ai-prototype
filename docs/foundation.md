# 工程地基与选型

## 项目目标

本项目定位为 ToB 后台管理系统的前端原型底座。当前阶段优先保证页面组织、组件体系、UI 规范和迭代边界稳定，后续可以逐步替换静态数据、接入真实接口、扩展权限和业务模块。

## 开发语言

选择 TypeScript + React。

原因：

- TypeScript 能在原型阶段就固定数据结构，减少后续从原型转工程时的返工。
- React 生态成熟，适合复杂表单、列表、详情页和多模块后台系统。
- 组件和业务逻辑可以按模块拆分，便于多人协作和持续扩展。

## 构建工具

选择 Vite。

原因：

- 启动快，适合高频调整的原型开发。
- 配置轻，后续可以自然接入环境变量、代理、构建优化和部署流水线。
- 与 React、TypeScript 和现代 CSS 的兼容性稳定。

## 组件选型

核心组件库选择 Ant Design 5。

使用边界：

- 表格、表单、筛选、弹窗、抽屉、分页、上传、日期选择等复杂业务组件优先使用 Ant Design。
- 图标优先使用 lucide-react，保证工具型按钮和导航图标风格一致。
- 通用布局、业务卡片、指标块等在 `components/` 中二次封装，避免页面直接堆砌重复样式。

## 状态与数据

- React Router 管理页面路由。
- TanStack Query 管理服务端状态，包括列表、详情、提交后的刷新和错误态。
- Zustand 管理轻量客户端状态，例如侧边栏折叠、用户偏好、原型模式开关。
- 原型阶段静态数据统一放在 `src/data/`，后续接入接口时保留页面结构，只替换数据来源。

## 模块扩展规则

新增业务模块时按以下结构放置：

```text
src/features/module-name/
  api.ts
  components/
  hooks.ts
  types.ts
```

路由页面放在 `src/pages/`，页面只负责组合模块能力，不承载过多业务细节。

## 决策原则

- 后台系统以效率、清晰、稳定为先，不做营销式首屏。
- 页面优先服务高频操作：筛选、比较、审批、编辑、导出、查看详情。
- 先统一布局、间距、密度、颜色和组件使用方式，再扩展业务模块。
- 新增抽象必须能减少重复或稳定边界，不能只为了“看起来更架构化”。
